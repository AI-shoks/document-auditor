# Eval harness — content-error precision/recall

This measures the quality of the **LLM content-auditing engine** numerically,
instead of eyeballing a couple of sample documents. It does NOT cover the
GOST formatting engine — that part is deterministic (no LLM) and is covered
by unit tests, not precision/recall.

**Out of scope:** completeness check (document vs. assignment/ТЗ — does the
document cover everything the assignment requires) is not evaluated here.
It needs a two-file input (document + assignment) instead of a single
document, which this harness doesn't model yet. Planned as a next
iteration, not part of this eval.

## Why a separate eval pipeline, not `audit.py` directly

`audit.py`'s production schema uses a free-text `location` field and an
unconstrained `type` string, because that's the most natural output for a
human reading a report. But free text can't be matched deterministically
against ground truth — the model's wording will never exactly match a
human's wording. So the eval harness uses its own schema (`schema.py`)
with:

- a closed `type` enum: `typo`, `grammar`, `factual`, `contradiction`, `logic`;
- a structural `paragraph` index instead of a free-text location, computed
  from the same paragraph-splitting function (`paragraphs.py`) used to
  build the numbered prompt (`[P1] ...`, `[P2] ...`) the model sees.

This makes matching a pure key comparison (`type`, `paragraph`) — never a
comparison of description text.

## Ground truth

Ground truth is authored by hand, **not generated by the model under
test** — otherwise the auditor would be graded against itself. Documents
are a mix of real (anonymized) reports and hand-written text with
deliberately planted errors of known type and paragraph. See
`docs/FORMAT.md` for the exact file format and `show_paragraphs.py` for
finding the right paragraph numbers.

Each `docNN.ground_truth.json` has a `status` field (`draft`/`ready`) so
`run_eval.py` can skip documents that aren't finished yet instead of
silently scoring them as zero-error documents.

## Matching algorithm

For each document:
1. Build the set of ground-truth keys `(type, paragraph)`.
2. Build the set of predicted keys `(type, paragraph)`.
3. Greedily consume: each predicted key that matches an unused ground-truth
   key is a **TP** (and that ground-truth entry is consumed, so duplicates
   are matched one-to-one, never double-counted). A predicted key with no
   matching unused ground-truth entry is a **FP**. Ground-truth entries left
   unconsumed at the end are **FN**.

`precision = TP / (TP + FP)`, `recall = TP / (TP + FN)`,
`f1 = 2PR / (P + R)`. Computed per document and as a micro-average
(summed TP/FP/FN across all 10 docs, then one P/R/F1) — more stable than
averaging 10 per-doc F1 scores, since one doc isn't penalized/rewarded out
of proportion to how many errors it actually contains.

Implementation: [`match.py`](match.py), unit-tested in
[`test_match.py`](test_match.py).

## Determinism caveat

The model is called with `temperature=0` for the most reproducible
behavior the API allows. This is **not a guarantee of bit-identical
output across runs** — Claude (like other LLMs) can still produce minor
variation at temperature=0 due to floating-point non-associativity in
batched inference. Treat the numbers below as representative of a single
run, not as a frozen constant.

## Prompt versioning

The system prompt is a parameter (`--prompt-version`), loaded from
`prompts/<version>.py` (a module exposing a `PROMPT` string). This lets
different prompt revisions be compared on the same dataset with the same
matching logic. Current baseline: `v1` (`prompts/v1.py`).

## Running

```
cd eval
python run_eval.py --prompt-version v1
```

Requires `ANTHROPIC_API_KEY` in `../.env` (same as the main app). Prints a
per-document table plus an aggregate row, and writes the same data to
`results.json`.

### Observed failure modes (from manually inspecting FP/FN on this run)

- **Category boundary ambiguity, not a missed error**: in doc06 and doc08
  the model found the right error at the right paragraph but labeled it
  `logic` where ground truth says `contradiction` (doc06 P4: "4C declared
  useless" — is that a logical leap or a contradiction with the source
  text? defensible either way), and `factual` where ground truth says
  `logic` (doc08 P4: payback period conclusion doesn't follow from the
  stated numbers). These count as one FP + one FN each under strict
  key matching, which is correct behavior for the matcher — it just means
  the 5-category taxonomy has genuine grey zones the prompt doesn't fully
  resolve.
- **Conflated findings** (doc10): the model merged two distinct planted
  errors (a roadmap step-order inconsistency and a separately-located
  "pilot now optional" contradiction) into one finding instead of
  reporting them at their two separate paragraphs — a genuine recall
  miss, not a ground-truth or harness artifact.
- One ground-truth authoring bug was caught and fixed during this run:
  doc10's paragraph numbers were off by one because a paragraph was
  missed when counting by hand instead of running `show_paragraphs.py`
  first. Underlines why `FORMAT.md` requires using that script rather
  than counting paragraphs by eye.
