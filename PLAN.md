# Document Auditor — карта проекта

Портфолио-проект для роли Applied AI / LLM Engineer.
Правило: один этап за раз, после этапа — рабочий артефакт + git commit.
Реализация каждого этапа — отдельная сессия, по явному запросу пользователя.
Карта ниже фиксирует маршрут целиком, но **не меняет порядок этапов на ходу** —
любые корректировки маршрута (упростить/выкинуть/переставить этап) решаются
между этапами, а не во время текущего.

## Этап 1 — Сквозной пайплайн (текст → Claude API → JSON)
- [x] Статус: ЗАВЕРШЁН
- Цель: доказать, что базовый цикл "документ → промпт → структурированный ответ" работает целиком.
- Артефакт: hello_world.py (вызов API) + audit.py (находит заложенные ошибки в sample.txt,
  отдаёт JSON {"errors":[...], "summary"}); репозиторий на GitHub (приватный).
- Done when: audit.py отдаёт валидный JSON с осмысленными ошибками по sample.txt, и это
  запушено в репозиторий. ("Все заложенные ошибки каждый раз" — не гейт: LLM недетерминирован,
  гоняться за идеальным повтором на каждом запуске — ловить флаки, а не мерить готовность.)
- Навыки роли: базовая интеграция с LLM API, парсинг ответов, промпт-инжиниринг верхнего уровня.

## Этап 2 — Pydantic-схема (AuditError, AuditReport)
- [x] Статус: ЗАВЕРШЁН
- Цель: заменить ручной разбор JSON на типизированную схему, зафиксировать контракт данных.
- Артефакт: модели AuditError и AuditReport на pydantic; audit.py валидирует ответ через них.
- Done when: невалидный ответ модели (например, лишнее поле или неверный тип) ловится
  pydantic-ошибкой, а не падает молча или с трудночитаемым traceback.
- Навыки роли: дизайн данных/контрактов, надёжность вокруг недетерминированного вывода LLM.

## Этап 3 — Поддержка форматов (.txt/.docx обязательно, .pdf опционально)
- [x] Статус: ЗАВЕРШЁН (.txt/.docx; .pdf не реализован — опционально по плану)
- Цель: принимать документы в реальных форматах, а не только sample.txt. Приоритет — .docx
  (реальный кейс с доками); .pdf — опциональная надстройка, не блокер этапа.
- Артефакт: функция extract_text(path) с веткой по расширению, минимум txt/docx.
- Done when: команда аудита успешно обрабатывает реальный .txt и реальный .docx и даёт
  осмысленный отчёт. .pdf — бонус, его отсутствие не блокирует переход к Этапу 4.
- Навыки роли: работа с реальными входными данными, инженерная часть до LLM-вызова.

## Этап 4 — FastAPI: POST /audit
- [x] Статус: ЗАВЕРШЁН
- Цель: превратить CLI-скрипт в сервис с HTTP-интерфейсом.
- Артефакт: FastAPI-приложение с эндпоинтом POST /audit, принимающим файл и возвращающим
  AuditReport (JSON).
- Done when: curl/Postman-запрос с файлом возвращает 200 и валидный AuditReport;
  есть базовая обработка ошибок (неподдерживаемый формат, пустой файл → понятный HTTP-код).
- Навыки роли: построение сервисного слоя вокруг LLM-логики — то, что отличает "скрипт" от "продукта".

## Этап 5a — Миграция на structured outputs / tool use
- [x] Статус: ЗАВЕРШЁН
- Цель: убрать самодельный воркэранд с ```json-оберткой, перейти на нативный механизм
  структурированного вывода Claude (tool use). Чистый рефакторинг, без новой функциональности.
- Артефакт: audit.py/сервис использует tool use вместо строкового парсинга JSON.
- Done when: парсинг JSON не зависит от того, обернула ли модель ответ в markdown;
  существующая функциональность (Этапы 1-4) работает как прежде.
- Навыки роли: продвинутые возможности API (tool use) — отличает уверенного Applied AI
  инженера от того, кто просто шлёт текстовые промпты.

## Этап 5b — Режим patch
- [x] Статус: ЗАВЕРШЁН
- Цель: добавить новую функциональность — предложение исправления, а не только диагноз.
- Артефакт: новый режим, возвращающий предлагаемый исправленный фрагмент текста
  в дополнение к описанию ошибки.
- Done when: patch-режим выдаёт применимое исправление хотя бы для одного класса ошибок.
- Навыки роли: продуктовое расширение LLM-фичи поверх готового структурированного вывода.

## Этап 6 — Eval harness (10 размеченных документов, precision/recall)
- [ ] Статус: не начат
- Цель: измерить качество аудитора численно, а не "на глаз".
- Артефакт: набор из 10 документов с заранее размеченными ошибками + скрипт,
  считающий precision/recall (или F1) по сравнению с ground truth.
- Done when: скрипт детерминированно выдаёт числовые метрики по всем 10 документам
  и их можно сравнить между промпт-версиями.
- Навыки роли: оценка качества LLM-систем — самый дефицитный и самый ценный для портфолио
  навык, показывает зрелость подхода (не "работает на одном примере", а "измерено").

## Этап 7 — Deploy + видео
- [ ] Статус: не начат
- Цель: сделать проект доступным и презентуемым без локального запуска.
- Артефакт: деплой на Railway/Render + 60-секундное демо-видео.
- Done when: сервис доступен по публичному URL и отвечает на реальный запрос;
  видео показывает весь путь от загрузки документа до отчёта.
- Навыки роли: умение довести проект до состояния "можно показать работодателю",
  а не оставить в виде локального скрипта.

## Критический разбор маршрута (без смягчений)

**Где реальный риск проекта:**
- Главный риск — не технический, а в Этапе 6. Без него весь проект остаётся
  "вайб-кодингом с LLM": работает на sample.txt, а дальше на словах. Если до Этапа 6
  дело не дойдёт (а по опыту таких маршрутов — велик шанс застрять на Этапе 4-5,
  потому что деплой и FastAPI визуально интереснее, чем разметка 10 документов руками),
  проект не закроет главный пробел в резюме: "умеет ли человек измерять качество LLM-системы".
- Этап 3 (форматы файлов) — наименее ценный для портфолио этап, но именно на нём
  легко завязнуть: .docx/.pdf парсинг полон мелких неприятностей (таблицы, сканы,
  битая кодировка), и это инженерная рутина, не имеющая отношения к Applied AI как роли.
  Риск — потратить день на парсинг PDF и решить, что "проект почти готов", хотя ключевая
  ценность (eval) ещё не начата.
- Этап 5 разбит на 5a (рефакторинг на tool use) и 5b (patch-режим как фича) именно
  потому, что это два разных артефакта — смешивать их в одном коммите противоречило бы
  правилу проекта "один этап = один артефакт".
- Подмена приватности репозитория (приватный вместо публичного) — разумное решение для
  работы с реальными/тестовыми данными документов, но если конечная цель — показать код
  работодателю, об этом придётся вспомнить отдельно на Этапе 7 (приватный репо не покажешь
  ссылкой в резюме без действия — нужно либо открыть, либо описать в видео/PDF).

**Что реально ценно для портфолио (по убыванию):**
1. Этап 6 (eval/метрики) — без вариантов, это то, что отличает кандидата, который "просто
   вызывает API", от того, кто понимает LLM-инжиниринг как инженерную дисциплину.
2. Этап 5a (structured output/tool use) — показывает знание актуального API, а не велосипедов
   на регулярках по markdown-обёртке. Этап 5b (patch) — приятный бонус сверху, но менее значим,
   чем 5a и тем более чем Этап 6.
3. Этап 4 (FastAPI-сервис) — стандартный, но необходимый сигнал "умеет делать сервис",
   без него это просто скрипт.
4. Этап 7 (deploy+видео) — ценно для презентации, но почти не показывает квалификацию
   как Applied AI/LLM Engineer — это больше DevOps-обвязка.
5. Этап 3 (форматы файлов) — наименее ценный с точки зрения LLM-навыков; можно упростить
   до одного формата (например, только .txt + .docx) без потери сигнала для резюме.
6. Этап 2 (Pydantic-схема) — гигиенический шаг, нужен, но сам по себе не история для резюме.

**Что можно упростить — но решать это отдельно, между этапами, не сейчас:**
- Этап 3: ограничиться .txt и .docx, выкинуть .pdf-парсинг (самый трудоёмкий и наименее
  показательный) либо отложить его как "опционально, если будет время".
- Этап 7: вместо полноценного деплоя с инфраструктурой достаточно простого варианта
  (Render free tier / Replit) — деплой здесь работает как доказательство "не только на
  моей машине", сложная инфраструктура тут избыточна.

**Важно:** эти упрощения — решения на будущее, а не правки маршрута прямо сейчас.
Текущий этап (Этап 2) доводится до коммита/push по плану как есть; пересмотр карты —
отдельный разговор после этого.

**Развилка после Этапа 4 — прочитать перед тем, как начинать Этап 5/6:**
Главный риск маршрута не технический, а поведенческий: FastAPI и деплой визуально
интереснее, чем разметка 10 документов руками, поэтому соблазн после Этапа 4 свернуть
в "ещё немного допилить сервис" вместо Этапа 6 — реальный. Этап 6 (eval/precision/recall) —
не опциональный пункт для галочки, а единственное, что отличает "вызываю API" от
"инженерю LLM-систему". Если на развилке после Этапа 4 возникнет идея пропустить или
отложить Этап 6 в пользу Этапа 5/7 — это тот самый паттерн, и его стоит замечать вслух,
а не поддаваться ему молча.

## Идеи на проверку после Этапа 6

Сюда складываются гипотезы об улучшении пайплайна, которые НЕ внедряются в MVP
по догадке "наверное, так лучше" — у них должна быть измеренная дельта precision/recall
от Этапа 6, прежде чем менять текущую реализацию.

- **Вход в Markdown вместо plain text** (mammoth / markitdown вместо текущего
  extract_text() на plain-тексте) — гипотеза: сохранение структуры документа
  (заголовки, списки, таблицы) при конвертации может улучшить качество поиска ошибок
  моделью по сравнению с потерей структуры в plain text. Проверять после Этапа 6:
  прогнать те же 10 размеченных документов через вариант с Markdown-конвертацией
  и сравнить precision/recall с текущим baseline (plain text). Менять пайплайн
  только если дельта измерена и положительна — не по умолчанию.

## Архитектурная идея на будущее (не для текущего MVP)

Гипотеза о целевой архитектуре аудитора, зафиксированная для памяти — не план
текущих этапов и не повод менять маршрут сейчас:

- **Два движка вместо одного**: детерминированный движок (проверка по ГОСТ —
  форматирование, структура — через python-docx, без LLM, без недетерминизма)
  + LLM-движок (смысловые/содержательные ошибки — то, что делает audit.py сейчас).
  Разделение важно, потому что ГОСТ-проверки должны быть 100% воспроизводимы
  (это не то, что стоит доверять LLM), а контентные ошибки — там, где LLM нужен.
- **Вход**: документ + критерии (а не просто текст документа) — критерии задают,
  что именно проверять (это сужает и параметризует задачу вместо одного фиксированного
  системного промпта на все случаи).
- **Основной формат — .docx** (не .txt) — реальный кейс это документы в Word,
  .txt в текущем MVP — суррогат для скорости первых этапов.

Эта идея не меняет текущий маршрут (Этапы 1-7 как есть) и не реализуется сейчас —
решение, встраивать ли её и когда, принимается после Этапа 6, а не как догадка
внутри MVP.
